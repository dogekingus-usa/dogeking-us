Proof-of-History Explained (Without the Jargon)

Proof-of-history (PoH) is Solana's signature innovation. To understand why it matters, you first need to understand the fundamental problem every blockchain faces: the timing problem.

Traditional blockchains like Bitcoin and Ethereum solve this problem through consensus mechanisms that force validators to communicate with each other to agree on a timestamp and order. Bitcoin's proof-of-work requires miners to solve computational puzzles, and Ethereum's proof-of-stake requires validators to cast votes. Both approaches are secure, but they're slow because every validator needs to talk to every other validator to establish time.

How PoH Flips the Problem Upside Down

    Basic Solana Terminology: The Glossary You Actually Need

    When you start exploring Solana, you'll encounter terms that don't exist on other blockchains. Here's what they mean:

    <<<<<<< HEAD

    SPL Token ¢â'¬âEUR The standard token format on Solana (equivalent to Ethereum's ERC-20). Any token you interact with on Solana ¢â'¬âEUR USDC, PYTH, JUP, BONK, or any meme coin ¢â'¬âEUR is an SPL token. Unlike Ethereum where each token has its own smart contract, SPL tokens use a unified program (Token Program) that manages all token balances.

    Sealevel ¢â'¬âEUR Solana's parallel smart contract execution engine. While Ethereum processes transactions one at a time, Solana can process non-overlapping transactions simultaneously. This is the technical breakthrough that gives Solana its speed advantage.

    Validator ¢â'¬âEUR A computer running Solana software that verifies transactions and maintains the blockchain. Validators are incentivized through staking rewards and transaction fees. There are approximately 1,500-2,000 active validators on Solana as of 2026.

    Leader ¢â'¬âEUR The validator currently responsible for producing blocks. Solana has a leader schedule (determined by stake weight) that rotates roughly every 4 slots (~1.6 seconds). During their leader slot, a validator collects pending transactions, orders them via PoH, and propagates the block.

    Slot ¢â'¬âEUR The smallest unit of time on Solana, approximately 400 milliseconds. Each slot corresponds to one opportunity to produce a block. A "confirmed" transaction has been included in a slot and validated by the network.

    Epoch ¢â'¬âEUR A period of time (approximately 2-3 days, ~432,000 slots) used for administrative functions like updating the validator set and distributing staking rewards. Staking rewards are calculated per epoch.

    Rent ¢â'¬âEUR Solana charges a one-time "rent" fee to store data on-chain (account balances, smart contract state, NFT metadata). This is because Solana validators need to maintain all account data in active memory (RAM) for fast access. You can reclaim most of this rent by closing the account. For most users with small token balances, rent is negligible (~$0.01-$0.05 per account).

    Compute Units (CU) ¢â'¬âEUR Similar to Ethereum's gas but measured differently. Each transaction has a compute budget (default ~200,000 CU) that limits how much computation it can consume. Complex transactions (e.g., swaps across multiple pools) may need more CU. Unlike Ethereum, unused CU is refunded, and CU prices are deterministic rather than auction-based.

    Priority Fee ¢â'¬âEUR An optional additional fee you can pay per compute unit to prioritize your transaction during network congestion. Because Solana fees are normally so low, priority fees remain insignificant ($0.001-$0.01 extra) even during busy periods ¢â'¬âEUR a stark contrast to Ethereum where priority fees can cost $50+.
